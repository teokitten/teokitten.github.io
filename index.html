case_01.md

CSPM Notification Automation – Scenario-Based Tutorial

First use-case format · Adopted org-wide
// challenge

In 2023, there was no established use case format in Dynatrace documentation. The challenge was to create one from scratch that could bridge the gap between product features and real customer problems – demonstrating value through a concrete, scenario-based narrative rather than a feature description, for a technically demanding security audience.

// approach

I chose a scenario-first structure, opening with a real operational problem: a security team receiving 400,000 daily AWS alerts uses Dynatrace to reduce that noise to a handful of actionable Jira tickets through automated filtering, enrichment, and routing. Before walking the reader through configuration, I included an explanation of how the mechanism works – so users understand the logic behind what they're setting up, not just the steps. The content then covers prerequisites, configuration steps, and a testing procedure – ending with the outcome of a successful setup.

// result

My first use case became the org-wide structural model for use case documentation across Dynatrace.

Security Posture Management – Documentation Architecture

Cross-functional · Built under ambiguity
// challenge

Security Posture Management (SPM) arrived as a new capability introduced through the Runecast acquisition, with no prior documentation or established structure. Covering it required simultaneous coordination across different engineering and product teams in an environment where scope, priorities, and requirements kept evolving until release.

// approach

I drove the entire documentation process: decided what and where needed to be documented, aligned cross-functional stakeholders across Kubernetes and AppSec SPM teams, proposed and iterated on the full documentation structure via PR, and coordinated the review cycle around the product's first release. I managed uncertainty by anchoring on what was known, prioritizing accordingly, and structuring the work to absorb last-minute changes. In parallel, I collaborated with developers and UX on in-product text, submitted bugs, and contributed improvement ideas before release.

// result

A complete SPM documentation set delivered from scratch in one and a half months, on top of regular commitments – on time, without compromising quality. This included the SPM capability (with a comprehensive FAQ), the KSPM setup and configuration, the Security Posture Management app, and a use case tutorial to demonstrate the capability's value through a practical, scenario-based experience.

case_03.md

Security FAQ – Multi-Platform Troubleshooting

Three audiences · Single source
// challenge

The Application Security FAQ is a comprehensive troubleshooting and guidance page covering all Security topics. It was meant to serve three distinct audiences simultaneously: customers on the latest Dynatrace platform, customers on the classic platform, and Managed customers – who ran on classic only. Troubleshooting steps and fixes often differed significantly between platforms, and in many cases entire topic areas applied exclusively to one or the other. A poorly structured FAQ risked overwhelming customers with content that didn't apply to them.

// approach

I built the FAQ from scratch and continuously improved it using real customer pain points identified through Zendesk support data. Rather than maintaining separate versions or cluttering the page with repeated caveats, I handled platform distinctions internally using conditional scripts and content placeholders – allowing the page to serve all three audiences from a single source.

// result

A structured, navigable FAQ covering ~50 questions across topics including detection, monitoring, configuration, attack blocking, data retention, consumption, and licensing. Improvements led to a measurable quarter-over-quarter reduction in related customer tickets.

process

Documentation philosophy

Documentation principles aren't found in style guides. They emerge from taking product quality personally.

01.
Documentation as part of the product

I think of documentation as part of the product, not a byproduct of it. My goal is to make complexity feel navigable – to write for the user who is frustrated, in a hurry, or seeing a concept for the first time.

02.
Early involvement

I join projects at the design and development stages. The earlier I'm involved, the better for everyone. I ask the right questions when changes are still easy to make, so gaps don't surface in production.

03.
Docs-as-code

Markdown, Git, pull requests, CI pipelines. Documentation managed like code doesn't drift from the product it describes – it gets reviewed, versioned, and shipped with the same rigor as any production artifact.

04.
AI tools, thoughtfully applied

I use AI tools to synthesize complex technical inputs, accelerate drafting, speed up repetitive tasks, and maintain consistency across large documentation sets. AI accelerates the work. Domain expertise determines what survives the edit.
